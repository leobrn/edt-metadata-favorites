# EDT Metadata Favorites — руководство разработчика

Документ описывает устройство плагина, ключевые технические ограничения и порядок проверки изменений.
Пользовательские сценарии приведены в [руководстве пользователя](user-guide.md).

## Технологический стек

- Eclipse/OSGi и Common Navigator Framework;
- API модели метаданных 1С:EDT;
- SWT/JFace для пользовательского интерфейса;
- Maven Tycho для сборки bundle, feature и p2-репозитория;
- JUnit 4 в headless OSGi runtime.

Плагин запускается на Java 17. Для сборки используется JDK 21 или новее и Maven 3.9.9 или новее.
Целевая версия платформы задаётся в `app/targets/default/default.target`.

## Модули

```text
app/
  bom/                                      общая конфигурация Maven/Tycho
  bundles/edt.metadata.favorites/           OSGi bundle плагина
  features/edt.metadata.favorites.feature/  устанавливаемая Eclipse feature
  repositories/edt.metadata.favorites.repository/
                                             p2 update site
  targets/default/                          target platform 1С:EDT
tests/edt.metadata.favorites.tests/          автоматические тесты
```

Основные файлы bundle:

- `META-INF/MANIFEST.MF` — идентификатор, версия, зависимости и execution environment;
- `plugin.xml` — команды, обработчики, меню, фильтр, декоратор и горячие клавиши;
- `plugin.properties` — пользовательские подписи;
- `build.properties` — состав собираемого bundle;
- `icons/` — изображения команд и декоратора.

Технический namespace проекта — `edt.metadata.favorites`.

## Архитектура

Пользовательские команды делегируют изменение состояния в `PinOperations`. Хранилище `PinStore`
не зависит от UI. После операции обновляются декоратор и дерево Навигатора.

```text
Контекстное меню ─┐
Горячая клавиша ──┼─> PinOperations ─> PinStore
Панель редактора ─┘         │
                            └─> обновление Навигатора и декоратора

Кнопка фильтра ─> ToggleFilterHandler ─> PinnedOnlyFilter ─> PinStore
Диалог ─> FavoritesManagementDialog ─> FavoriteTreeModel ─> PinStore
```

Граница интеграции с моделью 1С:EDT сосредоточена в `MetadataPinSupport`. Получение проекта, UUID,
FQN и обход вложенных объектов не следует дублировать в UI-классах.

## Карта исходного кода

Исходники находятся в `app/bundles/edt.metadata.favorites/src/edt/metadata/favorites/`.

| Класс | Ответственность |
|---|---|
| `Activator` | Жизненный цикл bundle, экземпляр `PinStore`, журналирование |
| `PinStore` | Загрузка, изменение и атомарное сохранение избранного |
| `PinnedObjectKey` | Ключ объекта: имя проекта и стабильный UUID |
| `PinTarget` | UUID объекта и читаемая FQN-метка |
| `PinOperations` | Единая точка выполнения пользовательских операций |
| `MetadataPinSupport` | Адаптация элементов Навигатора и EMF-модели 1С:EDT |
| `PinMenuContribution` | Динамическое контекстное меню |
| `PinToggleHandler` | Команда переключения состояния текущего элемента |
| `EditorPinToggleHandler` | Команда для активного редактора |
| `ManageFavoritesHandler` | Открытие диалога управления |
| `FavoritesManagementDialog` | SWT/JFace-диалог, поиск и применение изменений |
| `FavoriteTreeModel` | Построение независимой от UI модели дерева |
| `FavoriteTreeNode` | Узел дерева диалога |
| `ToggleFilterHandler` | Активация фильтра Common Navigator Framework |
| `PinnedOnlyFilter` | Правила видимости элементов и кэширование обхода дерева |
| `PinDecorator` | Отметка избранных элементов |

## Хранилище

Состояние находится в
`<workspace>/.metadata/.plugins/edt.metadata.favorites/pinned.yaml`.
Файл не входит в проекты конфигураций.

Ключ объекта состоит из имени Eclipse-проекта и UUID. FQN используется только как читаемая метка,
поэтому переименование объекта не меняет его идентичность.

Формат файла имеет собственную версию, не связанную с `Bundle-Version`. При изменении схемы необходимо:

1. повысить `CURRENT_FORMAT_VERSION`;
2. согласованно обновить чтение и запись;
3. определить поведение для предыдущих версий;
4. сохранить защиту от перезаписи файла более новой неизвестной версии;
5. добавить тест миграции или совместимости в `PinStoreTest`.

Запись выполняется через временный файл и атомарное перемещение, если оно поддерживается файловой системой.
После ошибки чтения состояние не должно сохраняться поверх исходного файла.

## Рекурсивное избранное

`PinStore` различает:

- индивидуально выбранные объекты;
- корни рекурсивно выбранных веток;
- исключения внутри рекурсивных веток.

При вычислении эффективного состояния приоритет имеет индивидуальный объект, затем ближайшее исключение
или рекурсивный корень в UUID-пути. Изменение этого порядка требует отдельных тестов для вложенных веток
и будущих потомков.

## Фильтрация и кэши

`PinnedOnlyFilter` сохраняет путь от корня дерева до избранного объекта. Проверка потомков рекурсивна,
поэтому используются два кэша с разным временем жизни:

- кэш наличия избранных объектов зависит только от `PinStore` и инвалидируется по `modCount`;
- кэш потомков зависит от текущей структуры Навигатора и живёт только один проход фильтрации.

Структура Навигатора может измениться без изменения `PinStore`: при загрузке, импорте, удалении объекта
или ленивом раскрытии узла. Поэтому кэш потомков нельзя делать долговременным или инвалидировать только
по `modCount`.

`ToggleFilterHandler` передаёт CNF полный набор активных фильтров. Передача только идентификатора
плагина отключит штатные фильтры пользователя.

## Расширение функциональности

Новая пользовательская команда обычно требует:

1. объявления команды и обработчика в `plugin.xml`;
2. подписи в `plugin.properties`;
3. реализации действия в `PinOperations`;
4. вызова обновления UI после изменения состояния;
5. автоматического теста логики;
6. ручной проверки доступности команды в реальном 1С:EDT.

Условия отображения контекстного меню остаются в `PinMenuContribution`, но изменение состояния должно
выполняться через `PinOperations` и `PinStore`.

Новые типы объектов метаданных добавляются через `MetadataPinSupport` и, при необходимости,
через справочники группировки в `FavoriteTreeModel`.

## Сборка и тестирование

Полная проверка:

```shell
mvn -f app/pom.xml clean verify
```

Комплект для GitHub Releases формируется из корня проекта:

```powershell
.\build-release.ps1
```

Скрипт повторяет полную проверку и создаёт `.build/releases/<version>/` с update-site ZIP,
bundle JAR, `release-notes.md` и `SHA256SUMS.txt`. Версия берётся из `app/pom.xml`, а текст изменений —
из секции `Unreleased` файла `docs/CHANGELOG.md`. Каталоги `target/` являются промежуточными и
не используются как публикуемый release.

Для повторной упаковки уже проверенных артефактов допустим
`.\build-release.ps1 -SkipBuild`. Перед публикацией этот режим используется только для той же
сборки и неизменённого дерева исходников.

Тестовый fragment `tests/edt.metadata.favorites.tests` запускается Tycho Surefire в headless OSGi runtime.
Тесты покрывают:

- сериализацию, совместимость формата и рекурсивное состояние `PinStore`;
- группировку и вложенность `FavoriteTreeModel`;
- критические идентификаторы и структуру деклараций `plugin.xml`.

Перед выпуском дополнительно проверяются в реальном 1С:EDT:

- установка из собранного p2-репозитория;
- команды Навигатора и редактора;
- горячие клавиши;
- фильтр и сохранение состояния после перезапуска;
- диалог управления на нескольких проектах;
- корректность работы после удаления или переименования объекта.

Полный порядок runtime-проверки приведён в
[чек-листе ручного тестирования](manual-testing-checklist.md).

## Требования к изменениям

Изменения должны сохранять разделение UI, операций и хранилища. Неочевидные обходы ограничений EDT
документируются рядом с кодом кратким комментарием, объясняющим причину или контракт, а не очевидное
действие.

Порядок подготовки изменений и требования к описанию ошибок приведены в
[CONTRIBUTING.md](CONTRIBUTING.md).
